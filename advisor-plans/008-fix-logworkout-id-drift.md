## Why this matters

When a user logs a set from the **manual "Registrar entrenamiento" picker**
(`LogWorkoutPage`) and PocketBase is available, the page fetches the
`exercises_catalog` collection and maps each record with `mapPBCatalog`, which
returns **`id: rec.id`** — the PocketBase record's *autogenerated 15-char random
primary key* (e.g. `k3n8x2q7m1p9z0a`). That value is then written verbatim into
`sets_log.exercise_id` (the column that is the **sole identity key** for all
score history, charts, volume, and PR detection).

Consequences, all silent:

- The **same physical exercise** logged from a program day (which logs the
  semantic id from `workouts.ts`, e.g. `pushup_std`) versus from this picker
  (which logs a random PB id) becomes **two unrelated score histories and two
  charts** that never merge.
- The random id matches **none** of the PR patterns, so PRs from the picker are
  never detected.
- The catalog already ships a human-meaningful **`slug`** field (kebab-case,
  unique, required) that is fetched on the very same record but **thrown away**.

This plan changes the picker to log a **stable, human-meaningful slug** instead
of the random id. It is a small, contained correctness fix.

> **Scope boundary (important — read before deciding the fix is "done"):** this
> plan makes the picker stop emitting *random* ids and start emitting *stable
> slugs*. It does **not** unify the two id namespaces in the codebase: the PB
> catalog uses **kebab-case slugs** (`knee-push-up`) while `workouts.ts` /
> programs / the bundled JSON use **snake_case ids** (`pushup_std`). So after
> this plan, a movement logged from the picker (slug) and the "same" movement
> logged from a program (snake id) **may still live under different ids**. Full
> cross-namespace unification is **Plan 011** (rebuild the bundled catalog from
> the seeds and pick one canonical scheme). 008 is still a strict improvement —
> *stable + meaningful* beats *random* — and is a prerequisite mindset for 011.

### The drift: `mapPBCatalog` discards `slug` and keeps the random `id` (VERBATIM, `LogWorkoutPage.tsx:71-73`)

```tsx
function mapPBCatalog(rec: any): CatalogItem {
  return { id: rec.id ?? '', name: rec.name ?? '', muscles: rec.muscles ?? '' }
}
```

`rec.id` is the PocketBase autogenerated primary key (random). `rec.slug` (the
intended stable identifier) is available on `rec` but never read.

### Where the PB catalog overwrites the local one (VERBATIM, `LogWorkoutPage.tsx:112-114`)

```tsx
const res = await pb.collection('exercises_catalog').getList(1, 200, { sort: 'name' })
if (!cancelled && res.items.length > 0) {
  setCatalog(res.items.map(mapPBCatalog))
```

When PB is available, the picker list is **entirely replaced** by `mapPBCatalog`
output, so every picked exercise carries the random `rec.id`. (When PB is *not*
available the picker falls back to `extractCatalog()`, lines 46–69, which uses the
semantic snake-case ids from `WORKOUTS` / `SUPPLEMENTARY_EXERCISES` / the bundled
JSON — those are already stable, so the fallback path is fine.)

### Where the picked id is written into permanent history (VERBATIM, `LogWorkoutPage.tsx:217-222`)

```tsx
setPromises.push(logSet(ex.id, workoutKey, {
  reps: set.reps,
  weight: set.weight ? parseFloat(set.weight) : undefined,
  rpe: set.rpe ? parseFloat(set.rpe) : undefined,
  timestamp: baseTs,
}, date))
```

`ex.id` is whatever `mapPBCatalog` (or `extractCatalog`) put on the item. This is
the value that lands in `sets_log.exercise_id`.

### Proof the `slug` field exists on every record (VERBATIM, `pb_migrations/1774000001_created_exercises_catalog.js:45-51`)

```js
{
  "autogeneratePattern": "",
  "hidden": false,
  "id": "text1001000002",
  "max": 0,
  "min": 0,
  "name": "slug",
  "pattern": "",
  "presentable": false,

```

`slug` is `required: true` and `type: "text"` → it is always present and
non-empty on fetched records, so `rec.slug` is safe to use as the identity.

### How `logSet` writes the id (VERBATIM reference, `useProgress.ts:289-294` — DO NOT EDIT here)

```tsx
await pb.collection('sets_log').create({
```

Confirms `exercise_id` is stored verbatim — there is no server- or hook-side
normalization that would rescue a bad id. The id you pass is the id forever.

### Step 2: Change `mapPBCatalog` to use the slug as the stable identity

Replace the function body so the picker's item id is the **slug**, falling back
to the record id only if a slug is somehow missing (defensive; `slug` is
`required` so this should never trigger, but it avoids ever producing an empty
id):

```tsx
function mapPBCatalog(rec: any): CatalogItem {
  // Identity MUST be the stable, human-meaningful slug — never the random PB
  // primary key (rec.id), which silently fragments score history. slug is a
  // required field on every exercises_catalog record (migration 1774000001).
  return { id: rec.slug || rec.id || '', name: rec.name ?? '', muscles: rec.muscles ?? '' }
}
```

Notes the executor must preserve:
- Keep `name` and `muscles` exactly as they were (`rec.name ?? ''`,
  `rec.muscles ?? ''`) — only the `id` source changes.
- Use `||` (not `??`) for the slug fallback so an empty-string slug still falls
  through to `rec.id` rather than logging `''`.
- Do **not** rename the `id` field on `CatalogItem` — downstream code (`ex.id`
  at line 217, the picker rendering, `extractCatalog`'s `seen` map) keys on `id`.
  You are changing the *value*, not the *shape*.
